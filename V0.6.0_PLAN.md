# v0.6.0 Plan — ctx-saver Token Reduction Release

**Type:** 🟢 Optimization release (no behavior break)
**Priority:** High (cumulative token savings + DX improvements)
**Estimated effort:** 3–4 days
**Target version:** v0.6.0
**Spec version:** 1

---

## Why this release

ctx-saver v0.5.x ทำงานถูกต้องและ ROI เป็นบวกสำหรับ heavy tasks (build/test/spec dump)
แต่จากการใช้งานจริงและ feedback จาก AI ผู้ใช้จริง 2 sessions พบว่า:

1. **Fixed overhead ~5,100 tokens/turn** จาก tool definitions ซึ่ง ~400-600 tokens เป็น
   freshness policy block ที่ซ้ำใน 5 retrieval tools — ลดได้ทันทีโดยไม่กระทบ behavior
2. **Round-trip overhead** — `auto_index_threshold_bytes = 5120` (5KB) ต่ำเกินไป
   ทำให้ไฟล์ Go ขนาดกลาง (200+ บรรทัด) ต้อง round-trip ผ่าน `output_id` แทนที่จะ
   return inline ตรงๆ
3. **No purge command** — ตอน switch context หรือ cache เน่า ไม่มีทาง reset cleanly
4. **Code files ใหญ่** ส่ง summary head/tail ดี แต่ AI มักต้องการแค่ structure
   (function/type signatures) — `--fields=signatures` ลด output token ได้ 80-90%

ทั้ง 4 จุดเป็น optimization ไม่ใช่ feature ใหม่ ship รวมเป็น v0.6.0 ได้ปลอดภัย

## Scope (4 features)

| #   | Feature                       | Token impact        | Risk |
| --- | ----------------------------- | ------------------- | ---- |
| F1  | Freshness policy dedup        | -400-600 / turn     | ต่ำ  |
| F2  | ctx_purge command             | (DX, ไม่ใช่ token) | ต่ำ  |
| F3  | Threshold tuning + config     | -1 turn / file read | ต่ำ  |
| F4  | Selective fields (signatures) | -80-90% / file read | กลาง |

# Feature 1 — Freshness policy deduplication

## Problem

`internal/server/server.go` ฝัง freshness policy block (5-7 บรรทัดต่อ tool) ใน
description ของ 5 retrieval tools: `ctx_search`, `ctx_list_outputs`,
`ctx_get_full`, `ctx_outline`, `ctx_get_section`

Total ~30-35 บรรทัด ≈ **400-600 tokens ส่งซ้ำทุก turn ทุก session ทุก project**

Evidence จาก AI ผู้ใช้จริง:

> "Freshness policy block ที่ซ้ำ 5 ครั้งใน tool description — เห็นมันทุก turn
> ในฐานะ AI ที่ใช้ tool รู้สึก verbose จริง"

## Solution

ย้าย full freshness policy ไปที่ `ctx_session_init` response (ส่งครั้งเดียวต่อ
session) ใน tool descriptions เหลือบรรทัดเดียวอ้างอิง

### Before (ใน 5 tools)

```text
CACHE FRESHNESS POLICY:
Each match includes freshness.stale_level = fresh | aging | stale | critical.
• fresh / aging — use data as-is.
• stale — warn the user; offer to re-run via ctx_execute.
• critical (>7 days) — DO NOT use for decisions; surface user_confirmation_required prompt first.
Heuristic: if the user asks for "ล่าสุด", "current", "latest", or "now" — call ctx_execute to re-run instead of returning stale cached data.
```

### After (ใน 5 tools)

```text
Returns freshness.stale_level field — see ctx_session_init for usage policy.
```

### Where full policy moves to

ใน `ctx_session_init` response, เพิ่ม field:

```json
{
  "freshness_policy": {
    "stale_levels": ["fresh", "aging", "stale", "critical"],
    "actions": {
      "fresh": "use as-is",
      "aging": "use as-is",
      "stale": "warn user; offer ctx_execute refresh",
      "critical": "DO NOT use; require user_confirmation_required prompt"
    },
    "refresh_keywords_th": ["ล่าสุด", "ปัจจุบัน"],
    "refresh_keywords_en": ["current", "latest", "now"]
  }
}
```

## Files to modify

- `internal/server/server.go` — ลบ freshness blocks จาก tool descriptions (5 จุด)
- `internal/handlers/session_init.go` — เพิ่ม `freshness_policy` field ใน response
- `internal/server/server_test.go` — update tests (เดิมเช็ค `"freshness"` ใน
  description; เปลี่ยนเป็นเช็คว่ามีการอ้างอิงถึง `session_init`)

## Risk & mitigation

**Risk:** AI ที่ไม่เคยเรียก `ctx_session_init` มาก่อนอาจไม่เข้าใจ stale_level

**Mitigation:**
- `ctx_session_init` เป็น tool แรกที่ทุก AI session ควรเรียก (already documented)
- บรรทัดอ้างอิงสั้นๆ ใน tool description บอกว่า "see ctx_session_init for policy"
- ถ้า AI เรียก tool โดยไม่เรียก session_init → ยังเห็น `stale_level` enum ใน
  response schema, action fallback คือ paranoid (treat as stale)

## Tests

- T1.1: Tool descriptions ทั้ง 5 tools ต้องไม่มีคำว่า "stale_level" ในรูปบรรยาย
  (มีแค่ใน reference line)
- T1.2: `ctx_session_init` response ต้องมี `freshness_policy` field
- T1.3: Total token count ของ tool definitions ต้องลดลง ≥ 300 tokens (วัดด้วย
  `tiktoken` หรือ approximation)

## Acceptance

- [ ] 5 tool descriptions มีบรรทัดอ้างอิงเดียว
- [ ] `freshness_policy` field ปรากฏใน `ctx_session_init` response
- [ ] Tests pass
- [ ] Manual: เรียก ctx_session_init แล้วเห็น policy ครบถ้วน

# Feature 2 — ctx_purge command

## Problem

ไม่มี clean way ที่จะ reset cached outputs ของ project ใช้กรณี:

- Switch context ใหญ่ (เริ่ม feature ใหม่ที่ไม่เกี่ยวกับเดิม)
- Cache เน่าหรือมี data เก่าที่รบกวน relevance score
- Pre-demo / pre-handover ต้องการ clean state

## Solution

เพิ่ม MCP tool ใหม่: `ctx_purge`

### Tool signature

```go
type PurgeInput struct {
    All     bool   `json:"all,omitempty"     jsonschema:"if true, also delete decision notes (ctx_note entries). Default false preserves notes."`
    Confirm string `json:"confirm"           jsonschema:"must be 'yes' to execute (safety check)"`
}

type PurgeOutput struct {
    OutputsDeleted int    `json:"outputs_deleted"`
    EventsDeleted  int    `json:"events_deleted"`
    NotesDeleted   int    `json:"notes_deleted"`
    NotesKept      int    `json:"notes_kept,omitempty"`
    Message        string `json:"message"`
}
```

### Behavior

| Flag                  | Cached outputs | Session events | Decision notes |
| --------------------- | -------------- | -------------- | -------------- |
| `confirm=yes` (default)  | ✅ ลบ          | ✅ ลบ          | ❌ เก็บ        |
| `confirm=yes, all=true` | ✅ ลบ          | ✅ ลบ          | ✅ ลบ          |

### Why notes preserved by default

Notes (`ctx_note`) เป็น decision log ที่มีค่าระยะยาว ลบยาก รอเอาคืน hard
กลับกัน cached outputs/events เป็น operational data ที่ regenerate ได้ตลอด

### Why require explicit confirm

ป้องกัน AI เผลอเรียก purge เพราะ tool name ตรงคำในประโยค user เช่น "purge old logs"

## Files to modify (NEW)

- `internal/handlers/purge.go` — handler ใหม่
- `internal/handlers/purge_test.go` — tests
- `internal/store/store.go` — เพิ่ม method `PurgeOutputs(ctx, projectPath) (int, error)`,
  `PurgeEvents(ctx, projectPath) (int, error)`, `PurgeNotes(ctx, projectPath) (int, error)`
- `internal/store/sqlite.go` — implement 3 purge methods (DELETE WHERE project_path = ?)
- `internal/server/server.go` — register ctx_purge tool

## Tool description

```text
[DESTRUCTIVE] Delete cached outputs and session events for this project.

Use when:
- Switching to a new feature/context that won't reuse current cache
- Cache became stale or noisy (irrelevant entries hurting search relevance)
- Pre-demo / pre-handover cleanup

Default behavior: deletes cached outputs + session events. Decision notes
(ctx_note entries) are PRESERVED.

Set all=true to also delete decision notes (irreversible — notes contain
long-term decisions worth keeping).

REQUIRES confirm="yes" to execute (safety check).
```

## Tests

- T2.1: `confirm=""` → error, no deletion
- T2.2: `confirm="yes", all=false` → outputs + events deleted, notes preserved
- T2.3: `confirm="yes", all=true` → all three deleted
- T2.4: Project isolation — purge ใน project A ต้องไม่กระทบ project B
- T2.5: Counts ใน response ต้องตรงกับจำนวนที่ลบจริง

## Acceptance

- [ ] Tool registered ใน server
- [ ] Default ลบ outputs + events เก็บ notes

# Feature 3 — Threshold tuning + configurability

## Problem

`config.go:111` กำหนด `AutoIndexThresholdBytes: 5120` (5KB) เป็น default

ผลข้างเคียง 2 ทาง:

**A. ไฟล์เล็กกว่า 5KB → return `direct_output` inline → ดี**

ปัญหาคือ feedback จริงระบุว่า **ไฟล์ Go ทั่วไป (200+ บรรทัด) เกิน 5KB ง่ายๆ**
ทำให้ตก path B

**B. ไฟล์เกิน 5KB → return `output_id` + summary → AI ต้อง round-trip เรียก
`ctx_get_full` ซ้ำ → +1 turn ต่อ file**

Evidence:

> "ctx_read_file output_id ต้องค้นไปสองครั้ง... ควรได้ content ตรง return ได้"

## Solution

2-part change:

### Part A: ขยับ default threshold

เปลี่ยน default จาก **5120 → 32768** (5KB → 32KB)

Justification:
- ไฟล์ Go ทั่วไป (300-500 บรรทัด) จะเข้า inline path
- 32KB เป็น size ที่ AI host รับ inline ได้สบาย (Claude Code/Copilot โดยทั่วไป
  รองรับ tool result หลายแสน tokens)
- ไฟล์ใหญ่จริง (build output, log dump, spec) ยังคงเข้า summary path ตามเดิม

### Part B: Document ใน .ctx-saver.yaml + guideline

เพิ่ม comment block ใน example config:

```yaml
summary:
  # Files/outputs smaller than this are returned inline (direct_output).
  # Larger ones are stored with an output_id and summarized.
  #
  # Tuning guideline:
  # - 32768 (default) — fits typical Go/Python files (300-500 lines)
  # - 65536 (aggressive) — keeps medium build outputs inline; uses more
  #   per-turn tokens but eliminates ctx_get_full round-trips
  # - 5120 (legacy v0.5.x) — minimal inline; forces output_id flow early
  auto_index_threshold_bytes: 32768
```

### Migration note

User configs ที่กำหนด `auto_index_threshold_bytes` ไว้แล้ว → ค่าเดิมยังใช้ได้
(no breaking change). User ที่ไม่ได้กำหนด → ได้ default ใหม่ 32KB

CHANGELOG ต้องบอกชัดว่า default เปลี่ยน เพราะอาจกระทบ "feel" ของ tool

## Files to modify

- `internal/config/config.go:111` — เปลี่ยน default value
- `internal/config/config_test.go:22` — update assertion
- `configs/freshness-examples/` — update example yaml
- `README.md` / `README.th.md` — document threshold tuning section
- `docs/migration-v0.6.md` — NEW migration note

## Tests

- T3.2: User-provided value ≠ 32768 ยังทำงานปกติ
- T3.3: ไฟล์ 30KB เข้า inline path (direct_output)
- T3.4: ไฟล์ 40KB เข้า summary path (output_id)

## Acceptance

# Feature 4 — Selective fields (`--fields=signatures`)

## Problem

ไฟล์ source code ขนาดกลาง-ใหญ่ ส่ง full content (หรือ head/tail summary) ไปให้ AI
แต่บ่อยครั้ง AI ต้องการแค่ structure: function signatures, type declarations,
constants — ไม่ต้องการ implementation body

**ลด output size ได้ 80-90% ต่อ file read** สำหรับ code-heavy files

Evidence:

> "ไฟล์ Go 300+ บรรทัด ยังส่ง summary head/tail แทน signatures ดี ได้
> condensed แล้ว แต่ถ้ามี --fields=signatures จะลดลงไป 90% เพิ่มเติม"

## Critical design decisions (จาก feedback validation)

### D1: Save full, filter on return

**ห้าม** save filtered content ลง DB เด็ดขาด เพราะถ้าเก็บแค่ signatures
→ `ctx_get_full output_id` จะได้ลำดับ line ไม่ตรงกับไฟล์จริง → AI patching จะพัง

**Implementation:**
- `ReadFile` handler save `FullOutput = full file content` ตามเดิม
- ก่อน return, ถ้า `--fields=signatures` → run extractor บน `rawOutput` →
  filter เป็น signatures-only view → return
- Cache key คงเดิม (`[read_file] <path>`) → request ที่สอง (ขอ full) จะ hit
  cache เดียวกัน, filter ตาม flag ของ request นั้น

### D2: Opt-in (ไม่ auto-apply)

V1 ทำเป็น opt-in flag เพื่อ:
- Validate pattern ทำงานถูกต้องกับ real files ก่อน
- AI ที่ไม่รู้จัก flag ก็ยังได้ behavior เดิม (no surprise)
- Cache key ไม่ต้องแยก variant

V2 อาจพิจารณา auto-apply สำหรับ code files > threshold หลัง v1 prove แล้ว

### D3: Line numbers preserved from original

Output ของ signatures ต้อง prefix ด้วย original line number เสมอ:

```text
[12]  func NewServer(cfg *Config) *Server {
[18]  func (s *Server) Start(ctx context.Context) error {
[34]  type Config struct {
[42]  func validate(c *Config) error {
```

ทำให้ AI เรียก `ctx_get_section` หรือ `ctx_get_full --line=18` ได้ตรงตำแหน่งจริง

## Tool signature change

```go
type ReadFileInput struct {
    Path          string `json:"path"`
    ProcessScript string `json:"process_script,omitempty"`
    Language      string `json:"language,omitempty"`
    Fields        string `json:"fields,omitempty" jsonschema:"optional view filter: 'signatures' returns only function/type/const declarations with original line numbers. Omit for full content. Supported languages: go, python, dart (basic regex; complex generics may be missed)."`
}
```

**Note:** `Fields` ตัวเดียวรองรับ `"signatures"` ใน v1; future values อาจเพิ่ม
`"imports"`, `"types"` แยก

## Architecture

NEW package: `internal/handlers/signatures/`

```
internal/handlers/signatures/
  detector.go      — extension → language map (.go → go, .py → python, .dart → dart)
  extractor.go     — generic line-filter + line number preservation
  patterns.go      — per-language regex sets
  patterns_test.go — pattern unit tests with real fixtures
```

### detector.go

```go
package signatures

import (
    "path/filepath"
    "strings"
)

type Language string

const (
    LangGo     Language = "go"
    LangPython Language = "python"
    LangDart   Language = "dart"
    LangNone   Language = ""
)

func DetectLanguage(path string) Language {
    ext := strings.ToLower(filepath.Ext(path))
    switch ext {
    case ".go":   return LangGo
    case ".py":   return LangPython
    case ".dart": return LangDart
    default:      return LangNone
    }
}
```

### extractor.go

```go
package signatures

import (
    "bufio"
    "fmt"
    "strings"
)

// Extract returns signatures-only view of source code with original line numbers.
// Returns ErrUnsupportedLanguage if lang has no patterns.
func Extract(content []byte, lang Language) (string, error) {
    patterns, ok := patternsByLang[lang]
    if !ok {
        return "", ErrUnsupportedLanguage
    }

    var out strings.Builder
    scanner := bufio.NewScanner(strings.NewReader(string(content)))
    scanner.Buffer(make([]byte, 1024*1024), 1024*1024) // support long lines

    lineNum := 0
    for scanner.Scan() {
        lineNum++
        line := scanner.Text()
        for _, re := range patterns {
            if re.MatchString(line) {
                fmt.Fprintf(&out, "[%d]  %s\n", lineNum, line)
                break
            }
        }
    }
    return out.String(), scanner.Err()
}
```

### patterns.go (Go — covered 100%)

```go
var goPatterns = []*regexp.Regexp{
    regexp.MustCompile(`^package\s+\w+`),
    regexp.MustCompile(`^import\s*\(`),
    regexp.MustCompile(`^func\s+`),       // covers methods (func (r *X) Foo)
    regexp.MustCompile(`^type\s+\w+\s+`),
    regexp.MustCompile(`^var\s+`),
    regexp.MustCompile(`^const\s+`),
}
```

### patterns.go (Python — covered ~95%)

```go
var pythonPatterns = []*regexp.Regexp{
    regexp.MustCompile(`^(async\s+)?def\s+\w+`),
    regexp.MustCompile(`^class\s+\w+`),
    regexp.MustCompile(`^\s+(async\s+)?def\s+\w+`), // class methods (indented)
    regexp.MustCompile(`^[A-Z_][A-Z0-9_]*\s*=`),    // constants by convention
    regexp.MustCompile(`^from\s+|^import\s+`),
}
```

### patterns.go (Dart — basic, document limit)

```go
var dartPatterns = []*regexp.Regexp{
    // Top-level: class, abstract class, mixin, enum, typedef
    regexp.MustCompile(`^(abstract\s+)?class\s+\w+`),
    regexp.MustCompile(`^mixin\s+\w+`),
    regexp.MustCompile(`^enum\s+\w+`),
    regexp.MustCompile(`^typedef\s+\w+`),
    // Top-level functions (best-effort: simple return types)
    regexp.MustCompile(`^[\w<>,\s\?]+\s+\w+\s*\(`),
    // Indented methods (inside class) — common case only
    regexp.MustCompile(`^\s+[\w<>,\s\?]+\s+\w+\s*\(`),
    // Constructors
    regexp.MustCompile(`^\s+(factory\s+)?\w+\s*\(`),
}
```

**KNOWN LIMITS for Dart (must document):**
- Generic return types ที่ซับซ้อน (`Future<Map<String, List<User>>>`) อาจ false negative
- Operator overloads (`operator ==`) อาจ false negative
- Multi-line signatures พลาด (regex line-based)
- Recommendation: ใช้ `process_script "grep -n 'def\|class\|...'"` สำหรับ
  complex Dart files

### YAML/JSON fallback

ถ้า detect ภาษาไม่ออก แต่ไฟล์เป็น `.yaml`/`.json` → fallback เป็น `ctx_outline`-style
(top-level keys + array structure) แทน error

ถ้า detect ไม่ออกเลย → return error: `"Fields=signatures unsupported for file
type .xyz; omit --fields or use process_script"`
